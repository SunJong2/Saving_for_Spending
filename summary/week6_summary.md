# 6주차 정리 — 프론트엔드 MVP 완성

홈 화면부터 시작해 백엔드의 모든 기능에 프론트를 연결한 주차.
**결과: MVP 완성** — 목표 생성 → 절약 기록 → 저금통 차오름 → 달성 축하 → 히스토리까지 완결된 순환.

## 0. 만든 것 한눈에 보기

```
index.html        홈 (진입 가드 + 저금통 진행률 바 + 기록 목록 + 수정/삭제 버튼
                      + 기록 상세/삭제 모달 + 목표 삭제 확인 모달)
add-goal.html     목표 생성 (중복 목표 시 홈으로 되돌림)
add-saving.html   절약 기록 (+ 축하 모달, 달성률 표시)
edit-goal.html    목표 수정 (기존 값 채우기 + 부분 수정)
history.html      달성 히스토리 (+ 목표별 기록 모달)

백엔드 개선
  PATCH /goals/current   target_amount 수정 추가 + current 이하 금지 검증
  POST /savings          달성 응답에 progress 추가
```

## 1. 이번 주 핵심 개념

### 진입 가드 패턴 (2단계)
```javascript
// 1차: 토큰 자체가 없음 → 조용히 로그인으로 (처음 방문일 수 있음)
if (!token) location.href = "login.html";

// 2차: 토큰은 있는데 무효(만료/위조) → 안내와 함께
if (response.status === 401) {
    localStorage.removeItem("token");        // 죽은 토큰 청소
    location.href = "login.html?expired=1";  // 쿼리 파라미터로 상태 전달
}
```
- 두 경우의 **의미가 다르므로 처리도 달라야 함** (만료 안내는 401에만)
- URLSearchParams로 페이지 간 상태 전달 — 페이지 이동 시 JS 변수는 사라지므로 URL을 매개로

### 화면별 가드 방향은 반대일 수 있다
```
add-goal    목표가 "있으면" 홈으로   (if res.ok)
add-saving  목표가 "없으면" 홈으로   (if !res.ok)
edit-goal   목표가 "없으면" 홈으로
```
같은 API(`GET /goals/current`), 반대 조건. 화면의 전제 조건에 따라 결정.
→ 백엔드 409/400은 데이터 방어(필수), 프론트 가드는 UX(권장). **목적이 달라 공존**

### fetch 심화
```javascript
// method 생략 = GET (기본값). 기본값 아닐 때만 명시 → 변경 요청이 눈에 띔
fetch(url, { headers: {...} })

// Content-Type은 "내가 보내는 body의 형식" 라벨
// → body가 있을 때만 필요 (GET/DELETE는 생략)
```
- 요청 헤더(내가 붙임)와 응답 헤더(서버가 붙임)는 별개
- `response`는 "왕복의 결과물 = 서버의 답장". `response.json()`은 그 봉투를 여는 것

### 응답 키 이름은 백엔드가 정한 것
```python
return {"history": [...]}   # 키는 DB 테이블명이 아니라 백엔드가 정한 계약
```
- `historyData.history` — 배열은 객체 안에 들어 있음
- 프론트는 DB를 전혀 모르고 **API 응답 형태만** 알면 됨 (백/프론트 분리의 실감)

### 진행률 바
```css
.progress-track { overflow: hidden; }        /* 채움이 트랙 밖으로 안 나감 */
.progress-fill  { transition: width 0.6s; }  /* 부드러운 애니메이션 */
```
```javascript
const fillWidth = Math.min(goal.progress, 100);   // 숫자는 실제값(103%), 바는 100까지
setTimeout(() => { fill.style.width = fillWidth + "%"; }, 100);
```
- **애니메이션이 안 보이는 이유**: 0%가 화면에 그려지기 전에 값이 바뀌면 transition할 구간이 없음 → setTimeout으로 한 박자 늦춰 0% → 실제값 전환을 만들어줌
- 4주차 "progress는 자르지 않고 바 그래픽만 100 제한" 결정이 CSS·JS 양쪽에서 구현됨

### 모달
```
평소 display: none → 켤 때 display: flex (CSS에 정의된 값으로 켜야 정렬이 살아남)
position: fixed + z-index → 스크롤 무관 화면 고정, 최상단
```
- 한 모달 안에 **여러 뷰**를 넣고 토글 (상세 ↔ 삭제 확인)
- **모달은 닫을 때 상태를 초기화** — 안 하면 다음에 열 때 이전 화면이 남음
- 되돌릴 수 없는 작업(삭제)은 반드시 확인 단계 + 결과를 미리 알려주기("기록도 함께 사라져요")

### 동적 요소에 이벤트 붙이기
```javascript
// map으로 만든 HTML 문자열 안에 onclick을 심고, 인자로 데이터 전달
`<div onclick="openGoalDetail(${h.id}, '${h.name}')">`   // 숫자는 그대로, 문자열은 따옴표
`<div onclick='openSavingDetail(${JSON.stringify(s)})'>` // 객체 통째로 (바깥은 홑따옴표)
```
- 4주차에 "id는 은닉이 아니라 응답에 포함"이라 결정한 것이 여기서 필수가 됨 (id 없으면 상세 조회 불가)

### 상태 공유용 변수
```javascript
let selectedSavingId = null;   // 모달 열기 함수와 삭제 함수가 공유
```
- `const`(재할당 불가) vs `let`(재할당 가능)
- 두 함수가 공유해야 하므로 **함수 밖**에 선언

### "함수 밖 영역"의 정체
```
함수 정의 = 레시피 적어두기 (실행 아님)
함수 밖의 줄 = 페이지 로드 시 흐름이 지나가며 실제 실행되는 코드
```
- 위→아래 실행 순서는 그대로. 함수 안은 호출될 때만 실행
- `<script>`를 body 맨 아래 두는 이유: **찾을 요소가 만들어진 뒤에 찾아야** 하므로
- 파이썬의 `def` 정의 후 맨 아래 호출 구조와 동일

### 부분 수정(PATCH) 프론트 처리
```javascript
const body = {
    name: nameInput || null,                          // 빈 문자열 → null
    target_amount: amountInput ? parseInt(amountInput) : null,
    deadline: deadlineInput || null
};
```
- **빈 입력창은 `undefined`가 아니라 빈 문자열 `""`** → 그대로 보내면 COALESCE가 빈 값을 저장해버림
- `값 || null`로 변환해야 "안 보냄 = 기존 값 유지"가 성립
- `.value = ...`로 입력창에 **기존 값 채우기** (읽기만 하던 것의 반대 방향)

## 2. 설계 판단 기록

### target_amount 수정 허용 + current 이하 금지 (C안)
- **왜 필요**: 목표 중심 앱에서 목표 금액을 못 고치는 건 치명적 결함. 포트폴리오에서도 지적당할 지점
- **왜 current 이하를 막나**: 가상 적립이라 "이미 모은 것보다 낮게" 설정할 현실적 이유가 없음. 허용하면 설정 즉시 달성되는 애매한 상태와 재판정 로직이 필요해짐 → **예외를 막아서 복잡성을 제거**
- 면접 답변 재료: 사용자 행동 근거 + 앱 본질(가상 적립) 이해 + 트레이드오프 판단

### 마감 연장 = deadline 수정
- 별도 기능이 아니라 edit-goal에 마감일 필드가 있으면 자동 해결
- 알림도 스케줄러 불필요 — `d_day < 0`을 프론트가 보고 안내 (게으른 판정)

### 홈 목록은 요약, 상세는 모달
- 메모를 목록에 다 노출하면 지저분, 안 보여주면 "배달음식"만 반복
- → **목록은 카테고리+금액, 탭하면 상세 모달(메모 전문 + 삭제)**
- 부수 효과: 흩어질 뻔한 "기록 삭제" 기능이 자연스러운 자리를 찾음 (목록에 삭제 버튼을 다닥다닥 붙이지 않아도 됨)

### 이미지 첨부 보류 확정
- 사용자가 누끼를 딸 리 없고, 스크린샷은 상태바가 노출됨
- 절약 기록에 음식 사진은 맥락상 어색
- → 백엔드 image_url 컬럼은 **유지**(퀵 버튼 아이콘 용도로 나중에 활용), 프론트만 미사용

### 삭제 후 갱신은 location.reload()
- 기록 삭제 시 목록과 진행률 바가 **동시에** 바뀜 → 부분 갱신보다 전체 새로고침이 확실하고 단순
- 목표 삭제도 동일 (새로고침하면 404 → "목표 만들기" 화면으로 자동 전환)

## 3. UI/UX 원칙 (이번 주 확립)

### 문구 규칙
```
바깥 화면(로그인/가입) → 감성 카피 ("모으는 재미, 쓰는 기쁨")
안쪽 기능 화면        → 질문형 제목 ("얼마를 아꼈나요?", "목표를 수정할까요?")
```
- 통일성은 문구가 같아서가 아니라 **말투(형식)가 같아서** 생김
- 화면마다 카피를 새로 짜낼 필요 없이 "사용자가 할 행동"을 질문으로 바꾸면 됨

### 정보 위계
- 모든 정보를 같은 크기로 나열하면 눈이 갈 곳을 잃음
- 히스토리 카드: 목표 이름(크고 굵게) > 금액(중간) > 달성 정보(작고 연하게)
- 축하 모달: "목표 달성!"이 주인공, "104% 달성"은 보조 → 크기로 구분
- 라벨("이름:", "금액:")은 없어도 알 수 있으면 제거

### 버튼 위계
```
주 행동   btn-primary (그라데이션)  — 절약 기록하기
보조 행동 btn-sub (회색)            — 목표 수정/삭제
위험 행동 btn-danger (빨강)         — 확인 모달 안에서만
```
- 삭제를 평소엔 조용하게(회색+빨간 글씨), 확인 단계에서 빨간 버튼으로 → 홈에서 시선을 뺏기지 않음

### 통일감 = 같은 역할엔 같은 수치
- 세 모달의 이모지 48px, 제목 margin `16px 0 8px`, 설명 margin-bottom 28px를 동일하게
- 값이 조금만 어긋나도 눈은 알아챔. `<br>` 하나로 세로 길이가 달라지는 것도 마찬가지
- 반복되는 인라인 스타일 → 나중에 CSS 클래스로 (**리팩토링은 반복이 다 드러난 뒤에 한 번에**)

### flex vs block
```
block  자식을 위→아래로 쌓음 (기본값)
flex   자식을 가로로 배치 + justify-content/align-items/gap 사용 가능
```
- 모달 오버레이가 flex인 이유: 카드를 화면 정중앙에 놓기 위해
- 켤 때 반드시 **원래 CSS에 정의된 display 값**으로

## 4. 겪은 이슈

1. **`historyData.goals` undefined** — 응답 키는 백엔드가 정한 `history`. 에러 메시지 `undefined is not an object (evaluating 'X.Y')` = "X가 undefined인데 Y를 꺼내려 함"
2. **HTML 태그 미닫힘** — `</div>` 누락으로 두 섹션이 중첩 → 하나 숨기면 둘 다 숨겨짐
3. **`justify-content: space-between` 오용** — 요소 2개일 땐 예뻤지만 4개가 되자 글자가 세로로 밀림 → 카드는 세로 흐름, flex는 필요한 줄에만
4. **PATCH에 빈 문자열 전송** — 이름을 비우면 `""`가 저장돼 이름이 사라짐 (금액은 NaN→null이라 우연히 정상 동작하는 불일치 상태)
5. **`conn.close` 괄호 누락** — 조용히 연결이 안 닫힘 (fetchone 때와 같은 유형, 에러가 안 나서 더 위험)
6. **모달 상태 초기화 누락** — 삭제 확인 화면인 채로 닫으면 다음에 열 때 그 화면이 뜸
7. **콘솔의 `Failed to load resource: 404`** — 코드가 처리 중인 예상된 응답이면 무시 가능. **JS 에러(Uncaught TypeError 등)와 구분할 것**
