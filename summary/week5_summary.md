# PART 1. 설계 회의 (자소서·면접 재료)

### 초과 달성 금액: 이월하지 않음
- 이월하려면 "목표에 소속되지 않은 돈(지갑)" 개념이 필요 → 모든 금액이 goal_id에 소속되는 현 구조와 충돌
- 이 앱에서 금액의 의미 = "어떤 목표를 향한 노력의 기록"이지 계좌 잔고가 아님 (가상 적립 컨셉)
- 초과분은 사라지는 게 아니라 이전 목표 히스토리에 "103% 달성"으로 남음 — 손실이 아니라 훈장
- progress는 100으로 자르지 않고 실제 값 표시. 100 제한이 필요한 건 프론트의 진행률 바 그래픽뿐 (데이터는 백엔드, 표현은 프론트)

### 기록 수정/삭제: 삭제만 구현, 수정은 생략
- 퀵 버튼(원터치 등록) 계획상 오입력 가능성 높음 + 금액 오류는 달성 판정까지 왜곡 → 삭제는 필수 탈출구
- 범위 제한: 진행 중 목표의 기록만 삭제 가능, 완료 목표의 기록은 동결 ("끝난 역사" 정책과 일관)
- 수정(PATCH)은 생략: 기록은 작고 단순한 데이터라 지우고 재작성 비용이 거의 0. 모든 자원에 CRUD 풀세트가 필요한 건 아니며 자원 성격에 맞게 결정
- 삭제 시 연쇄 효과: current_amount 차감을 같은 트랜잭션으로

### 마감 초과: 실패 처리 대신 연장 선택권 (연기)
- 절약 앱이 사용자를 혼내면 이탈함 → 부드러운 톤 유지
- 구현 힌트: 스케줄러 불필요. GET /goals/current의 d_day가 음수면 프론트가 연장 여부를 물으면 됨 (게으른 판정). 연장 = 이미 만든 PATCH의 deadline 수정

### 비밀번호/아이디 찾기: 미구현 (의도적)
- 비밀번호는 해시 저장이라 "찾기"는 원리상 불가능, 실체는 "재설정"
- 재설정은 이메일 발송 인프라(외부 서비스, 토큰 테이블, 만료 처리)가 필요한 별도 챕터 → 투자 대비 효과 낮음
- README에 "향후 과제: 이메일 인증 기반 재설정"으로 — 해시 원리를 안다는 어필이 됨

# PART 2. 백엔드

## 새로 배운 개념

### IDOR과 소유권 검사 (이번 주 최대 수확)
- goal_id/saving_id는 전역 순번(모든 사용자 공유) — 다른 계정을 만들어 직접 실험으로 확인
- 클라이언트가 id를 지목하는 API(`/goals/{goal_id}/savings`)는 남의 id를 넣는 공격이 가능 → **id로 접근하는 모든 API에 user_id 소유권 검사 필수**
```sql
SELECT id FROM goals WHERE id = ? AND user_id = ?   -- 없으면 404
```
- 거부는 403이 아닌 **404**: 403은 "존재는 한다"는 정보를 흘림. 로그인 에러를 뭉뚱그린 것과 같은 원리
- **"id 은닉은 보안이 아니다"**: 접근 제어는 검사가 하는 것. id는 프론트가 상세 조회/삭제에 필요하므로 응답에 포함하는 게 맞음 (히스토리에 goal_id, 기록 목록에 saving_id)
- 같은 자원은 같은 응답 모양으로 (두 기록 목록 API의 일관성)

### 환경변수와 .env
- SECRET_KEY 하드코딩 → 유출 시 누구나 토큰 위조 가능
- python-dotenv: `load_dotenv()` → `os.getenv("SECRET_KEY")`
- **.env는 반드시 .gitignore에** (코드는 공개, 비밀은 환경에만)
- 키 생성: `python -c "import secrets; print(secrets.token_hex(32))"`
- getenv는 실패해도 None 반환 (조용한 오작동) → 시작 시점에 None 검사로 크게 터뜨리기 ("조용한 오작동보다 시끄러운 실패")
- 키 교체 = 기존 토큰 전체 무효화 (옛 키로 서명된 토큰은 검증 실패) — 직접 확인함

### 날짜 다루기
- **"날짜 차이"는 시간 빼기가 아니라 날짜 정규화 후 빼기**: timedelta.days는 24시간 단위 경과라 마감 전날 23시 달성이 0일이 됨 → `.date()`로 시분초 제거 후 비교하면 보정값 없이 의미가 맞음
- strptime은 문자열 전체가 형식과 정확히 일치해야 함 (`unconverted data remains` 에러)
- utcnow() deprecated → `datetime.now(timezone.utc)` (naive datetime의 시간대 오해 문제)
- 서버 시간은 UTC가 관례: 만료 판정 등은 절대 시점이어야 하며, 현지 시간 변환은 표시할 때만

### 기타
- 존재 확인 vs 상태 확인: 존재가 궁금하면 조건 걸고 fetchone None 체크(SELECT 1), 상태가 궁금하면 그 컬럼을 SELECT해서 파이썬 if
- 검증 순서: 입력 검증(400)이 DB 조회(404)보다 먼저 — "문지기는 문 앞에"의 자연스러운 결과
- ORDER BY 없는 SELECT는 순서 미보장 (DELETE 후 반환 순서 바뀌는 것 직접 목격). 순서가 의미 있으면 반드시 명시
- 커밋 메시지: "git log에서 한 줄로 파악되는가" 기준. 유지보수 커밋은 '왜'(deprecated 대응 등)를 남기기

## 겪은 버그 (실전 기록)

1. **DELETE WHERE 조건 복붙 사고** — 목표 삭제에서 복사한 `WHERE goal_id = ?`를 그대로 둬서 기록 하나가 아니라 그 목표의 기록 전체 삭제. 코드는 같아도 문맥이 바뀌면 의미가 바뀜
2. **fetchone 괄호 누락** — `if cursor.fetchone is None:`은 함수 객체 비교라 영원히 통과 못 함 → 404가 죽고 None 언패킹으로 500
3. **fetchone 이중 호출** — 호출마다 다음 행을 가져오므로 두 번째는 None. 한 번만 호출해서 변수로 받고, 검사도 언패킹도 그걸로

# PART 3. 프론트엔드

## 디자인 시스템
- 참고: 토스/브랜디(흰 배경, 큰 타이포, 미니멀) + 그라데이션 포인트
- 뼈대는 미니멀, 포인트 컬러(보라→파랑 그라데이션)는 버튼/진행률/저금통에만
- CSS `:root` 변수로 박제 → 한 곳 수정으로 전체 반영, 모든 페이지가 style.css 하나 공유
- 가입/로그인은 같은 레이아웃 유지가 정석 (같은 패밀리 화면), 차별화는 타이틀 문구로 — 토스 방식

## 새로 배운 개념

### fetch로 POST 보내기
```javascript
fetch("/login", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ email, password })   // JS 객체 → JSON 문자열 (json.dumps 격)
})
```
- `response.ok`: 상태 코드 200번대면 true → 성공/실패 분기
- `{ email, password }`는 `{ email: email }`의 축약

### localStorage
- 브라우저 내장 저장소. **도메인별** 저장 (탭별 아님 — 모든 탭이 공유, 브라우저 재시작에도 유지)
- 토큰 보관 = /docs에서 메모장에 복사해두던 것의 자동화. "한 번 로그인하면 유지"의 원리
- 서버와 무관하게 클라이언트에 존재. 배포해도 같은 구조로 동작 (배포 도메인의 저장소에 각자 저장)
- 개발자 도구 → 저장 공간(Storage) 탭에서 확인/삭제

### 프론트 검증 vs 백엔드 검증 (판단 기준 확립)
```
데이터 유효성/무결성 (금액>0, 날짜 형식, 이메일 중복, 빈 값 가입 차단)
  → 백엔드 필수. 뚫리면 잘못된 데이터가 저장됨
입력 편의 (비밀번호 재확인, "입력해주세요" 실시간 표시, @ 형식)
  → 프론트 전용. 뚫려도 사용자 본인 오타일 뿐 데이터는 무결
```
- 비밀번호 재확인은 서버로 보내지 않음 (서버는 비밀번호 하나만 필요)
- 프론트 검증도 fetch 앞에서 return — 프론트판 "문지기는 문 앞에"

### JS 문법 조각
- `!==` / `===` 엄격 비교만 사용 (`==`는 타입 자동 변환으로 버그 온상)
- `!str`: 빈 문자열은 거짓으로 평가
- `str.includes("@")`: 포함 여부
- `location.href = "..."`: 페이지 이동
- id는 대소문자 구분 (errorMsg ≠ errormsg — 실제로 겪음)
- input type은 정해진 값만 (text, password, email...)
- form 태그 안 버튼은 기본 submit(페이지 이동) 동작 → div 사용으로 회피

## 유령 계정 사건 (이번 주 최대 디버깅)

**증상**: 빈 입력으로 로그인해도 index로 이동, 토큰 생성

**수사 과정에서 배운 것**:
1. 네트워크 탭에 요청이 **없는 것도 정보** — 뭐가 일어났는지만이 아니라 일어났어야 할 게 안 일어난 것을 찾기
2. "토큰이 생겼다 = setItem 실행 = response.ok = 200" — 결과물에서 역추적하는 사고
3. **진범: /docs 테스트 시절 빈 값으로 만들어진 email="" 계정** — 빈 입력 로그인이 그 계정으로 "정상 성공"한 것
4. 교훈: **"테스트하며 흘린 데이터가 유령이 되어 돌아온다"**, "프론트 버그처럼 보여도 원인은 데이터일 수 있다" — 코드만 보지 말고 데이터도 조사
5. 조사 도구: sqlite3 직접 SELECT/DELETE (지울 땐 기본키로: `WHERE id = 4`)

**재발 방지**: 백엔드 signup에 빈 값 검증(`.strip()` — 공백만 입력도 차단) + 프론트 양쪽 검증

**부수 수확 — 브라우저 캐시**:
- "코드는 맞는데 동작이 다르면, 실행 중인 코드가 내가 보는 코드가 아닐 가능성부터"
- 사파리가 옛 버전 HTML을 메모리 캐시에서 서빙 → 수정이 반영 안 된 것처럼 보임
- 대응: 강력 새로고침(Cmd+Option+R), 개발 중엔 네트워크 탭 "캐시 비활성화" 상시 체크
- 프론트 디버깅 표준 루틴: 상태 초기화(토큰 삭제) → 재현 경로 통제(새 탭, 직접 진입) → 콘솔+네트워크 관찰
